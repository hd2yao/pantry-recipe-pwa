# 食材管家 PWA 设计方案

## 背景和目标

为个人手机使用构建一款无需上架 App Store 的 PWA。用户把网页添加到手机主屏幕后，可以像使用普通 App 一样记录买入、消耗和剩余食材，并根据现有食材前往小红书或 B 站搜索菜谱。

第一版的真实目标不是做一套复杂的菜谱平台，而是解决三个高频问题：

1. 我现在还有什么菜？
2. 哪些菜已经放了较久，应该优先吃？
3. 用现有食材可以搜什么菜谱？

## 已确认约束

- 只供个人使用。
- 主要在手机上使用。
- 不上架 iOS App Store。
- GitHub 仓库公开。
- 第一版保持简单，不接入需要账号 Cookie 的抓取服务。
- 数据默认只存放在当前设备的浏览器中。

## 方案比较

### 方案 A：纯前端 PWA，本地保存（推荐）

- 优点：开发和部署最简单；不需要服务器、登录、数据库或 App Store；隐私边界清晰。
- 缺点：换手机不会自动同步；清除浏览器数据会丢失记录；后台通知能力有限。
- 适用：个人第一版，先验证是否愿意每天记账。

### 方案 B：PWA + 云端数据库

- 优点：可以跨设备同步、备份和定时推送。
- 缺点：需要账号体系、云数据库、隐私和运维成本，首版复杂度明显增加。
- 适用：确认长期使用后作为第二阶段。

### 方案 C：原生或侧载 iOS App

- 优点：系统通知、相机和后台能力更完整。
- 缺点：证书、签名、安装维护和 iOS 开发成本更高，即使不上架也不够省事。
- 适用：PWA 无法满足系统能力时再考虑。

## 推荐设计

采用方案 A：React + TypeScript + Vite 构建移动优先 PWA，使用浏览器本地存储保存数据，部署为静态网站。后续如果确实需要多设备同步，再将存储层替换成云端实现，页面和领域逻辑不需要推翻。

## 第一版范围

### 1. 今日总览

- 显示现有食材数量。
- 显示需要优先处理的食材。
- 显示临期和可能过期的食材数量。
- 提供“买了什么”和“做饭消耗”两个主操作。

### 2. 买入食材

- 记录名称、数量、单位、分类、购买日期和预计可保存天数。
- 对常见分类提供默认保存天数，用户可以覆盖。
- 同名、同单位食材可以合并数量，保留更早的购买日期作为提醒依据。

### 3. 记录消耗

- 从现有库存中选择食材和数量。
- 可选填写菜名或用途。
- 消耗后自动扣减库存；数量归零时从当前库存移出。
- 保存一条历史记录，便于回看今天做了什么。

### 4. 库存与保鲜状态

- 按“尽快吃、还新鲜、已过建议期、未设置”显示状态。
- 支持按状态和分类筛选。
- 显示购买日期、已存放天数和剩余建议天数。
- 保鲜状态只做提示，不把日期推断包装成食品安全结论。

### 5. 菜谱搜索

- 用户勾选现有食材，自动生成“食材名 + 菜谱”关键词。
- 一键打开小红书搜索结果。
- 一键打开 B 站搜索结果。
- 第一版不抓取、缓存或重排第三方平台内容。

### 6. 数据保护

- 自动保存到当前浏览器。
- 支持导出 JSON 备份。
- 支持导入由本应用导出的 JSON。
- 数据结构带版本号，为后续迁移预留空间。

## 非目标

- 不做账号、家庭共享或多设备同步。
- 不自动识别购物小票或拍照识菜。
- 不直接抓取小红书/B 站内容。
- 不做营养计算、热量管理或医疗建议。
- 不承诺浏览器关闭后的定时系统通知。
- 不根据日期单独判断食物一定能吃或一定不能吃。

## 信息架构

底部使用三个主入口：

1. `今天`：总览、临期提醒和快捷操作。
2. `库存`：完整食材列表、筛选、编辑和消耗。
3. `菜谱`：选择食材并前往平台搜索。

设置和数据备份放在右上角菜单中，不占用主导航位置。

## 数据模型

```ts
type PantryItem = {
  id: string;
  name: string;
  quantity: number;
  unit: string;
  category: string;
  purchasedAt: string;
  shelfLifeDays: number | null;
  createdAt: string;
  updatedAt: string;
};

type ConsumptionRecord = {
  id: string;
  dishName: string | null;
  consumedAt: string;
  lines: Array<{
    itemId: string;
    itemName: string;
    quantity: number;
    unit: string;
  }>;
};

type PantrySnapshot = {
  version: 1;
  items: PantryItem[];
  consumptionRecords: ConsumptionRecord[];
};
```

## 数据流

```text
表单输入
  -> 领域校验
  -> 买入/消耗规则
  -> 更新内存状态
  -> 写入本地存储
  -> 总览、库存、菜谱选择同步刷新
```

菜谱搜索只读取用户已选择的库存名称，编码为搜索关键词后打开官方网页搜索入口。2026-07-14 已验证以下入口可以接受关键词并返回网页：

- `https://www.xiaohongshu.com/search_result?keyword=...`
- `https://search.bilibili.com/all?keyword=...`

## 设计契约

### 任务模型

- 用户应能在 20 秒内完成一次买入记录。
- 用户应能在 30 秒内完成一次多食材消耗记录。
- 首页打开后应能一眼看到最该先吃的食材。

### 布局模型

- 移动端单列布局，主内容最大宽度 720px。
- 顶部为页面标题和数据菜单，底部为固定三栏导航。
- 主操作使用底部弹层或全屏表单，保证单手触达。
- 在 375px、768px 和 1440px 宽度下不得出现横向滚动。

### 风格模型与 Design Lock

- 视觉方向：清爽、生活化、可靠，不做营销型首页。
- 主色：蔬菜绿；预警使用琥珀色，过期建议使用克制的红色。
- 卡片只用于聚合真正相关的信息，避免装饰性卡片堆叠。
- 字号、间距和按钮触控区域优先服务手机使用。
- 不使用与食品状态无关的渐变、玻璃拟态或大面积装饰插画。

### 状态覆盖

- 首次使用空状态。
- 有库存的正常状态。
- 临期和超过建议保存期状态。
- 表单校验失败状态。
- 本地数据损坏、导入失败和存储写入失败状态。
- 平台搜索按钮不可用时保留可复制的搜索关键词。

## 错误和安全边界

- 数量必须大于 0，消耗数量不得超过库存。
- 本地数据解析失败时不覆盖原始内容，并显示恢复提示。
- 导入前校验版本和字段，确认后整体替换当前数据。
- 所有“过期”文案表述为“超过建议保存期”，并提示同时检查气味、颜色、质地和实际储存条件。
- 第三方平台链接使用新窗口打开，并带 `noopener,noreferrer`。

## 验收标准

- AC-001：可以新增食材，刷新页面后数据仍存在。
- AC-002：可以记录一个或多个食材的消耗，库存数量正确扣减并生成历史记录。
- AC-003：首页按当前日期正确显示已存放天数和优先级。
- AC-004：可以从现有库存选择食材，并打开带正确关键词的小红书和 B 站搜索页。
- AC-005：可以导出并重新导入完整数据。
- AC-006：应用可安装为 PWA，断网后仍能打开已缓存的应用壳。
- AC-007：375px、768px、1440px 下无横向溢出，主要触控目标不小于 44px。
- AC-008：核心领域规则有自动化测试，关键用户流程有浏览器测试。

## 后续阶段

只有第一版实际使用一段时间后仍有明确需要，才进入下一阶段：

1. 云端备份与跨设备同步。
2. 系统通知或每日提醒。
3. 常用食材快捷录入和语音输入。
4. 经平台允许的菜谱聚合或智能推荐。
